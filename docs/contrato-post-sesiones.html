3 · Dónde está escrito el schema de variables

NO está en el código. Está en la base de datos.

El schema de qué variables acepta cada plantilla vive en la tabla formato_campo, filtrada por el id_formato de la plantilla. El código solo la lee:

  • Query: getFormatoCampos() en agenteVoz.model.js:61-70 → SELECT nombre_campo, etiqueta, tipo_dato, requerido, longitud FROM formato_campo WHERE id_formato = ?
  • Cadena: plantilla 139 → id_formato 72 → filas de formato_campo.
  • La bandera de obligatoriedad es la columna requerido (1 = obligatorio).

👉 Para cambiar qué campos existen o cuáles son obligatorios, se edita la tabla formato_campo en la BD — no se toca código ni se redespliega.

5 · Cómo se valida

validarVariables(campos, variables) — agenteVoz.model.js:114-123:

const faltantes = campos
  .filter((c) => Number(c.requerido) === 1)          // solo los obligatorios
  .filter((c) => {
    const v = variables[c.nombre_campo];
    return v === undefined || v === null || v === ""; // ausente o vacío
  })
  .map((c) => c.nombre_campo);
return { ok: faltantes.length === 0, faltantes };

Es decir: solo comprueba que los campos requerido=1 estén presentes y no vacíos. No valida tipos, ni longitud, ni rechaza campos desconocidos.

6 · ¿Qué pasa si mandas algo que NO está mapeado?

Campo extra que no existe en el formato (ej. "COLOR_FAVORITO": "azul")
→ Se ignora en silencio. No da error. validarVariables solo itera sobre los campos del formato, así que ni lo mira. Se mergea en varsPrompt (sesiones.controller.js:110), pero si el prompt no lo referencia con {{COLOR_FAVORITO}}, queda inerte. La llamada procede normal.
Falta un campo obligatorio (ej. sin OFERTA_CREDICASH)
→ Rechazo 400 con { "codigo": "variables_incompletas", "faltantes": ["OFERTA_CREDICASH"] }. La sesión no se crea, no hay llamada. sesiones.controller.js:65-68
Falta un campo opcional que el prompt SÍ referencia (ej. el prompt usa {{correo}} y no lo mandaste)
→ No rechaza, pero queda "sin resolver": warning plantilla con N variable(s) sin resolver y el agente podría leer el placeholder en voz alta o dejar el hueco. sesiones.controller.js:132 (variablesSinResolver)
Tipo equivocado (ej. lead_id como string aunque en BD sea integer)
→ Se acepta. La validación no revisa tipo_dato ni longitud — son informativos. Target, al postear directo, controla el tipo, pero da igual porque no se valida.
